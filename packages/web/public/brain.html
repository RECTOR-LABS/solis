Signal Architecture
4-Layer Signal Detection
The core thesis: social buzz precedes dev activity (social), developers build first (leading), capital flows in (coincident), then market prices it (confirming). Each layer has different predictive power and lead time.
üê¶
Layer 0: Social Signals
Opt-in • earliest signal
"When Crypto Twitter starts buzzing about a topic, it often precedes developer activity by days or weeks."
LunarCrush v4
GET /public/coins/list/v2?filter=solana + /public/topic/:topic/v1
  • 24h social interactions + delta (z-scored)
  • Sentiment score (0-100)
  • Social dominance (% of crypto social volume)
  • Galaxy Score (composite 0-100)
  • Contributors + post counts per topic
  • Enabled via ENABLE_SOCIAL_SIGNALS flag

LLM Analysis
Two-Phase LLM Intelligence
SOLIS makes exactly 2 LLM calls per report. Both use Claude Haiku 4.5 via OpenRouter at temperature 0.3 with JSON schema enforcement for deterministic, parseable output. GLM-4.7 serves as fallback on 5xx errors.
üß†
LLM Call #1

Narrative Clustering

Receives condensed scored signals (top 20 social coins if enabled, top 30 repos by commits, top 15 programs by tx count, top 20 tokens by 14d price change, all anomalies, stablecoin flows) plus previous report narratives for continuity tracking.

// Input (condensed) { "social": { topByInteractions[], anomalies[], topBySentiment[] }, // if enabled "github": { anomalies[], newRepoClusters[], top30Repos[] }, "defi": { tvlTotal, tvlDelta, tvlAnomalies[], top10Dex[], stablecoinFlows }, "onchain": { top15Programs[] }, "market": { top20Tokens[], trending[], categoryPerformance[] }, "previous_narratives": [...] // continuity context } // Output per narrative { "name": "Meme-Centric DEX Dominance", "stage": "GROWING", "momentum": "accelerating", "confidence": 92, "social_signals": [...], "leading_signals": [...], "coincident_signals": [...], "confirming_signals": [...], "related_repos": [...], "related_tokens": [...], "related_protocols": [...] }
